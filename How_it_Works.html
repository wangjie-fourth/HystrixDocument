How it Works

流程图

下图展示了Hystrix调用依赖服务请求时，发生的事：

  • 1、构造HystrixCommand或者HystrixObservableCommand对象
  • 2、执行命令
  • 3、判断该请求的响应是否被缓存
  • 4、断路器是否开启
  • 5、线程池、队列、信号量是否饱和
  • 6、执行HystrixObservableCommand.construct()或HystrixCommand.run()
  • 7、计算断路器的健康程度
  • 8、执行降级逻辑
  • 9、返回成功响应

1、构造HystrixCommand或者HystrixObservableCommand对象

第一步就是构造出HystrixCommand或者HystrixObservableCommand对象来表示请求依赖服务的请求。向构造函数传递发出请求时所需的任何参数。

如果依赖服务返回的单一响应，就构造出HystrixCommand对象。比如：

HystrixCommand command = new HystrixCommand(arg1, arg2);

如果期望依赖项返回一个发出响应的Observable，则构造一个HystrixObservableCommand对象。 例如：

HystrixObservableCommand command = new HystrixObservableCommand(arg1, arg2);

2、执行命令

通过使用Hystrix命令对象的以下四种方法之一，可以执行命令的四种方法（前两种仅适用于简单的HystrixCommand对象，而不适用于HystrixObservableCommand）：

  • execute()：阻塞，然后返回从依赖项接收到的单个响应（或在发生错误的情况下引发异常）
  • queue()：返回一个Future，您可以使用它从依赖项中获得单个响应
  • observe()：订阅表示来自依赖项的响应的Observable，并返回复制该源Observable的Observable
  • toObservable()：返回一个Observable，当您订阅它时，它将执行Hystrix命令并发出其响应
K             value   = command.execute();
Future<K>     fValue  = command.queue();
Observable<K> ohValue = command.observe();         //hot observable
Observable<K> ocValue = command.toObservable();    //cold observable

同步调用execute()调用queue().get(). queue()依次调用toObservable().toBlocking().toFuture()。 也就是说，最终每个HystrixCommand都由Observable实现提供支持，即使是那些旨在返回单个简单值的命令也是如此。

3、请求响应是否被缓存

如果这个命令的请求是允许使用请求缓存，且这个请求的响应此时也是可用的，那么这个缓存的响应将立马以Observable形式返回。

4、断路器是否打开？

在你执行命令时，Hystrix将会检查断路器是否打开。

如果断路器打开，则Hystrix就不会执行该命令，将会路由到第8步去执行降级逻辑。

如果断路器未打开，就会执行第5步，检查服务是否有能力去执行该命令。

5、线程池、队列、信号量是否已经满了

如果执行命令相关联的线程池、队列（如果不是使用线程执行，那就是信号量）已经满了，则Hystrix不会执行该命令，而会路由到第8步去执行降级逻辑。

6、HystrixObservableCommand.construct()或HystrixCommand.run()

在这里，Hystrix通过以下方法调用对依赖项的请求：

  • HystrixCommand.run()：返回单一响应或者返回异常
  • HystrixObservableCommand.construct()：返回一个发射响应或发出onError通知的Observable对象

如果run()或者construct()方法执行超时，则线程将抛出TimeoutException（如果命令不是在单独线程执行的话，则一个单独的计时器线程抛出）。在这种情况下，Hystrix将会把结果路由到第八步执行降级逻辑，如果run()或者construct()无法被取消或者中断，则该方法返回的值将会被丢弃。

请注意没有任何方法能让潜在的线程强制停止工作，Hystrix在JVM能做的就是抛出InterruptedException。如果Hystrix所包装工作类不遵循InterruptedException，即使客户端已经收到TimeouteException，Hystrix线程中的线程还是会继续工作。尽管负载已“正确释放”，但此行为可能会使Hystrix线程池饱和。大部分Java的Http库不会中断InterruptedException。所以请正确配置HTTP客户端上的链接和读、写超时时间。

如果这个命令没有返回任何异常，并且返回一个结果，Hystrix会执行一些日志记录和指标报告后返回此响应。这种情况执行run()，Hystrix将返回一个发射单一响应的Observable对象，然后发出onCompleted()通知；如果是construct()，Hystrix将返回同样的Observable对象。

7、计算断路器的健康

Hystrix将健康、失败、拒绝、超时的请求状态数据发送给断路器，这些数据用来计算一系列的计算counter。

它使用这些统计信息来确定断路器何时应打开，在该时间点它会将随后的所有请求短路，直到经过恢复期为止，在此之后，在首先检查某些运行状况检查之后，它将再次关闭断路器。

8、降级逻辑

当命令执行失败时，Hystrix会尝试将错误转换到降级逻辑上：

  • 在第6步，执行run()、construct()方法抛出异常
  • 在第4步，断路器处于打开状态时
  • 在第5步，命令线程池、队列、信号量处于饱和状态时
  • 或者在执行命令时，超时了

编写降级逻辑，以从内存中缓存或通过其他静态逻辑提供通用响应，而无需任何网络依赖性。如果你非要在降级逻辑中使用网络调用，你应该使用另一个HystrixCommand或者HystrixObservableCommand。

在HystrixCommand情况下，你需要实现一个返回单一值的HystrixCommand.getFallback()的降级逻辑。

在HystrixObservableCommand.resumeWithFallback()，你需要实现一个返回Observable对象的HystrixObservableCommand.resumeWithFallback()来实现降级逻辑。

如果降级方法返回一个响应值，则Hystrix会把这个响应值返回给调用者。在HystrixCommand.getFallback()下，它会返回一个发射从方法接受值的Observable对象。在HystrixObservableCommand.resumeWithFallback()会返回相同的Observable对象。

如果尚未为Hystrix命令实现降级方法，或者降级方法本身引发异常，则Hystrix仍返回一个Observable对象，但它不发出任何内容并立即以onError通知终止。 通过此onError通知，将导致命令失败的异常发送回调用方。 （实施一个可能会失败的降级方法，这是一个糟糕的做法。您应该实现一个不可能失败的降级方法。）

不同方式调用Hystrix命令，其失败或者不存在降级方法的返回值也会不同：

  • execute()：抛出异常
  • queue()：成功返回Future对象，但是如果调用get()方法时，Future对象就会抛出异常
  • observe()：返回一个Observable对象，当你订阅它时，它将通过调用订阅者的onError方法立即终止
  • toObservable()：返回一个Observable，当您订阅它时，它将通过调用订阅者的onError方法终止

9、返回成功的响应

如果Hystrix命令执行成功，则会返回响应或者以Observable对象返回给调用者。这取决于你在第2步如何执行命令，Observable在返回你之前，会经过以下转变：

  • execute()：跟.queue()方法一样会返回一个Future()对象，然后在从Future上调用get()方法就会返回Observable发射出的单一值
  • queue()：将Observable转换到BlockingObservable，这样它就可以转换成Future对象，然后就返回这个对象
  • observe()：在开始执行命令的流程之前，就订阅Observable； 返回一个Observable，当您订阅该Observable时，将会发出通知
  • toObservable()：返回一个未被改变过的Observable对象；你必须在执行命令之前就订阅它

断路器

下图展示了HystrixCommand、HystrixObservableCommand与HystrixCircuitBreaker对象是如何交互的、逻辑流程和如何做决定的，以及断路器内部counter的行为。

断路器打开、关闭的详细描述：

  • 假设断路器上大量的请求超过了阈值，HystrixCommandProperties.circuitBreakerRequestVolumeThreshold()
  • 假设错误百分比超过阈值错误百分比，HystrixCommandProperties.circuitBreakerErrorThresholdPercentage()
  • 然后，断路器的状态从CLOSE到OPEN
  • 当断路器处于OPEN状态时，断路器会拦截所有的请求
  • 过了一段时间后HystrixCommandProperties.circuitBreakerSleepWindowInMilliseconds()，下一个单个请求让它通过（此时断路器处于HALF-OPEN状态）。如果这个请求失败，断路器转换到OPEN状态；如果这个请求成功，断路器就转换到CLOSE状态。

隔离

Hystrix使用隔板模式将依赖关系彼此隔离，并限制对其中任何一个的并发访问。

线程和线程池

客户端在各自的线程执行任务。这样可以将它们与调用线程（Tomcat线程池）隔离，以便调用者可以避免花费太长时间的依赖项调用。

Hystrix对每个依赖服务使用单独的线程池，这样即使该服务的线程池达到饱和的状态，也只会影响该服务自己的线程池。

如果不是这种线程池技术也是可以避免失败，但这要求你一直充分信任所有的客户端，并且能在失败后，快速响应。

Netflix设计Hystrix时，采用线程、线程隔离技术来达成隔离的主要原因有：

  • 许多应用程序针对由许多不同团队开发的数十种不同服务执行数十种（有时甚至超过100种）不同的后端服务调用
  • 每个服务提供自己客户端库
  • 客户端库可能总是在改变
  • 客户端可能会新增调用逻辑
  • 客户端可能包含重试、解析、缓存等其他行为的逻辑
  • 客户端库往往像是黑盒的：用户对其实现细节，网络访问模式，配置默认值等不透明
  • 在真实生产环境下，可能会出现调整参数值、或者客户端调整了某些行为，比如说：接口的QPS值
  • 即使客户端本身没有变化，服务本身也会发生变化，这可能会影响性能特征，进而导致客户端配置无效
  • 传递依赖关系可能会引入其他意外的客户端库，这些客户端库可能不是预期的，也可能配置不正确
  • 大多数网络访问是同步执行的
  • 异常和延迟也可能在客户端代码中发生，而不仅仅是在网络调用中

线程池的好处

在服务自己线程池使用线程隔离的好处：

  • 该应用程序受到完全保护，可防止客户端库失控。 给定依赖库的线程池饱满的话，而不会影响应用程序的其余部分
  • 可以让应用在引入新的服务时，降低风险。如果新的服务发生异常了，我们立马隔离这个服务，这样就不会影响整个应用了

这个好处其实很有意思。这样的话，就可以做到不影响线上运行

  • 当发生故障的客户端再次恢复正常运行时，线程池将被清除，应用程序将立即恢复运行正常的性能，而整个Tomcat容器不堪重负则需要很长时间才能恢复
  • 如果客户端配置错误的话，我们可以通过急剧增加的错误、延迟、超时、拒绝等数量表现出线程池的健康情况。然后你可以通过近乎实时动态配置来解决这些问题，这样就可以在不影响程序功能的情况下处理错误。
  • 如果客户端服务调整了性能属性（这通常都是一个问题），那么对应服务就需要调整属性（比如增加、减少超时时间；改变重试次数）。这里就可以通过调整线程池的属性（错误、延迟、超时、拒绝）来解决这种问题，而且还不会影响客户端、请求、用户的操作。

这个意思就是指当下游服务降低接口的QPS时，你再给Hystrix中这个服务的线程池提供那么多的线程其实是无用的。这个时候，你自己服务可以通过修改这个 线程池的属性来控制自己服务对依赖服务的QPS

  • 除了隔离优势之外，拥有专用的线程池还可以提供内置的并发性，可以利用这些并发性在同步客户端之上构建异步操作（类似于Netflix API如何在Hystrix命令上构建异步的Java API）

简而言之，线程池提供的隔离允许客户端库和子系统性能特征的不断变化和动态组合得到优雅处理，而不会造成中断。

注意：尽管使用单独线程提供隔离特性，但是您的客户端代码也应该具有超时、对线程中断响应的功能，这样就不会无限期阻塞住线程池，从而导致线程池资源耗尽。

线程池的缺点

线程池最主要的缺点就是增加了额外的计算开销。在每个线程执行命令时都可以涉及到队列、定时、上下文切换。

Neyflix在设计这个系统的时候，觉得接受这种代价来换取优点，并且认为这种缺点并不会对性能有太多影响。

线程的开销

Hystrix在子线程上测试执行construct()、run()方法的延迟，也会收集父线程整体方法的执行时间。这样你就可以看到Hystrix的开销（线程，指标，日志记录，断路器等）。

Netflix的API使用线程隔离技术每天处理上百亿的命令。每个API实例拥有40多个线程池，每个线程池有5～20个线程（大多数设置为10）。

下图展示了一个HystrixCommand在单个API实例上以每秒60个请求的速度执行（每个服务器每秒约350个线程在执行）

在中位数（或更低）处，拥有一个单独的线程是没有成本的。

在第90个百分点，拥有一个单独的线程要花费3ms的时间。

在第99个百分位数处，拥有一个单独的线程要花费9ms。 但是请注意，成本的增加远远小于单独线程（网络请求）的执行时间的增加，后者从2跳到28，而成本从0跳到9。

对于这样的电路，这种开销在90％或更高的百分比上被认为对于大多数Netflix用例都是可以接受的，以实现所具有的弹性。

对于包装延迟非常低的请求的电路（例如那些主要访问内存缓存的请求），开销可能会过高，在这种情况下，您可以使用其他方法，例如可尝试的信号量，尽管它们不允许超时， 提供最大的弹性优势，而没有开销。 但是，总的来说开销很小，以至于Netflix实际上通常比这种技术更喜欢使用单独线程的隔离优势。

信号量

你不仅可以使用线程池、队列数来限制对给定依赖服务的并发调用数，还可以使用信号量（或者计数器）达到这种目的。这使Hystrix无需使用线程池就可以减轻负载，但它不允许超时和退出。 如果您信任客户端，并且只想减少负载，则可以使用这种方法。

HystrixCommand和HystrixObservableCommand在下面俩个地方支持信号量：

  • Fallback：当Hystrix执行降级逻辑时，它总是在Tomcat线程池上调用？？？
  • Exection：如果你将execution.isolation.strategy设置成SEMAPHORE，则Hystrix将会使用信号量而不是线程池来限制调用该命令的并发父线程的数量。

你可以通过动态属性配置配置信号量，然后让其限制有多少的线程可以执行。您应该使用与确定线程池大小时类似的计算来确定信号量的大小（一个以亚毫秒的时间返回的内存中调用可以执行超过5000rps，信号量只有1或2，但默认值是10）

注意：如果一个依赖服务使用信号量完成隔离，然后它又变得不可见了，则父线程会一直保持阻塞直到网络调用超时。

一旦信号量达到限制，信号量就开始执行拒绝测量，但是为信号量填充的线程不可以消失。

请求合并

您可以在创造HystrixCommand之前，放置一个请求折叠的代码（抽象父级的HystrixCollapser）,将多个请求折叠放到一个请求调用里。

下图展示在俩种情况下，多个线程和网络调用的流程。假设俩者在10ms内请求都是并发的，第一个场景是没有请求合并的情况下，第二个场景是有请求合并的情况下。

为什么使用请求合并？

使用请求合并可以在并发执行HystrixCommand时，有效的减少线程数、网络连接时间。请求合并是以一种自动化的方式完成，不会强制代码库的开发人员手动编码来完成请求合并。

整体上下文（跨越所有的Tomcat线程）

理想情况下，请求合并是在整个应用的level来完成的。这样在Tomcat线程上来自任何用户的请求都可以合并到一起。

例如，如果你配置HystrixCommand在任何用户下都支持批处理请求来请求获取电影评级的服务依赖。当在同一个JVM下的任何用户发出这样的请求时，Hystrix会这个请求与其他人同样的请求放在一个请求合并的网络调用中。

注意，合并器会将单个HystrixRquestContext合并到一个网络调用中，所以下游系统必须支持这个场景下的处理。

用户请求上下文（单个Tomcat线程）

如果你配置HystrixCommand只对单个用户进行批处理的话，Hystrix会在单个Tomcat线程中合并请求。

比如，如果一个用户想为300个视频对象获取书签，Hystrix可以将他们合并到一个请求，而不是发出300个网络调用。

对象建模和代码复杂性

有时，你想创建一个对于消费者非常符合逻辑的对象模型，但这往往很难复合对象生产者的合理资源利用。

比如，一个300个电影对象的集合，现在遍历这个集合，然后在为每个对象调用getSomeAtrribute()方法来完善这个对象模型。如果简单这样实现，会导致在毫秒内调用300个网络请求，这很可能导致目标服务资源饱和。

有一些手工方法可以处理这种问题。比如，在用户调用getSomeAttribute()之前，要求他们一次性执行获取属性的视频对象，这样就可以一次性提取他们。

或者，你可以拆分这个对象模型。让用户必须一个地方获取视频列表，在另一个地方获取这个视频列表的属性。

这些方法也能解决性能问题，但是在某些情况下，会导致API和对象模型不能达到理想的模型、在使用模式上也可能不匹配。当多个开发人员在一个代码库上工作时，它们还可能导致简单的错误和效率低下，因为针对一个用例进行的优化可能会因另一个用例的实现和代码的新路径而中断。

但是将合并请求逻辑放到Hystrix层，无论如何创建对象模型，以什么顺序进行调用，其他开发人员是否知道需要优化，都无关紧要。

可以将getSomeAttribute()方法放在最合适的位置，并以适合使用模式的任何方式调用它，然后请求合并将自动将调用批量放入时间窗口。

请求合并的代价

请求合并的唯一代价就是在真正执行命令之后，会多一段延迟时间。最大延迟时间就是批处理的时间间隔大小。

如果你执行一个命令要花费5ms，批处理的时间间隔是10ms，在最坏的情况下，执行时间将花费15ms。通常，一个请求刚准备发送时不可能是正好要发送批处理的时候，通常是批处理间隔的一般，这时候执行命令久得话费10ms。

确定此成本是否值得取决于所执行的命令。 高延迟命令不会受到少量额外平均延迟的影响。 同样，给定命令的并发量很关键：如果很少有超过1个或2个请求被一起批处理，那么付出代价是没有意义的。 实际上，在单线程顺序迭代中，请求合并将是主要的性能瓶颈，因为每次迭代将等待10ms的批处理窗口时间。

但是，如果某个特定命令被大量同时使用，并且可以将数十个甚至数百个呼叫分批处理，则由于Hystrix减少了所需的线程数和与之连接的网络数，因此实现的吞吐率通常远远超过成本。

请求合并流程

请求缓存

HystrixCommand和HystrixObservableCommand可以定一个缓存key，可以用来在高并发环境剔除重复的请求。

这是一个示例流程，涉及HTTP请求生命周期和两个在该请求中工作的线程：

请求缓存的好处是：

  • 不同的代码路径可以执行Hystrix命令，而无需担心重复的工作

这在大型项目开发中很有好处。

比如，需要获取用户的Account对象的所有代码路径都可以这样请求

Account account = new UserGetAccount(accountId).execute();

//or

Observable<Account> accountObservable = new UserGetAccount(accountId).observe();

Hystrix RequestCache将只执行一次基础run()方法，并且尽管实例化了不同的实例，但执行HystrixCommand的两个线程仍将接收相同的数据。

  • 整个请求中的数据检索都是一致的

而不是每次执行命令时都可能返回不同的值（或回退），而是对同一请求内的所有后续调用进行缓存并返回第一个响应。

  • 消除重复的线程执行

由于请求缓存位于construct()或run()方法调用的前面，因此Hystrix可以在导致线程执行之前将重复数据删除。

如果Hystrix没有实现请求缓存功能，则每个命令都需要自己在Construct或run方法内部实现它，这会将其放在线程排队和执行之后。
